# Diagnosis — 20260415-1201 / Label-driven CI release pipeline + e2e gating

## Two gaps, one shared primitive

Both halves of this task ride the same signal: **does the merged (or merging) PR carry a `release:major|minor|patch` label?** Bundling them avoids two label-detection implementations.

| Half | Trigger | Effect |
| --- | --- | --- |
| **A. Release flow** | `pull_request: closed` to `main` with `merged == true` AND any `release:*` label | Run `npm version <type>`, push commit + tag, draft a GitHub release for the tag |
| **B. E2E gating** | `pull_request` (any) | Run e2e suite **only** if PR is labeled `release:*` (or push-to-main) — otherwise unit tests + lint + secrets only |

The existing `publish.yml` is the third hop. It already triggers on `release: published`, runs `bun run test && bun run build`, then `npm publish` with `NPM_TOKEN`. The draft → human-publishes path keeps the npm push behind a manual confirm without re-introducing the version-picking toil. Per `DO NOT`: `publish.yml` is not touched.

## Half A — `bump.yml` (new file)

### Trigger

```yaml
on:
  pull_request:
    types: [closed]
    branches: [main]
```

`pull_request: closed` fires on **both** merged and non-merged closes. Job-level `if:` filters to merged-only:

```yaml
if: |
  github.event.pull_request.merged == true &&
  (contains(github.event.pull_request.labels.*.name, 'release:major') ||
   contains(github.event.pull_request.labels.*.name, 'release:minor') ||
   contains(github.event.pull_request.labels.*.name, 'release:patch'))
```

This satisfies acceptance crit "Workflow only runs on merged PRs, not closed-without-merge" and "Defaults to patch if a release-labeled PR has ambiguous level" (the level-determination step picks the highest precedence label found, falling back to `patch`).

### Permissions + secrets

```yaml
permissions:
  contents: write   # push commit + tag, create release
```

No new secret needed — `GITHUB_TOKEN` (auto-injected) gets `contents: write` from the job-level grant. `gh` CLI on the runner reuses it. `NPM_TOKEN` stays solely on `publish.yml`.

### Checkout

```yaml
- uses: actions/checkout@v5
  with:
    ref: main
    fetch-depth: 0
    token: ${{ secrets.GITHUB_TOKEN }}
```

`ref: main` because for `pull_request: closed` the default ref is the merge-commit, which doesn't have a branch context for `git push`. We need to be on `main` to commit + push.

### Determining bump level

Single-shell determine step (precedence: major > minor > patch; default patch):

```yaml
- id: level
  env:
    LABELS: ${{ toJSON(github.event.pull_request.labels.*.name) }}
  run: |
    if   echo "$LABELS" | grep -q '"release:major"'; then echo "level=major" >> "$GITHUB_OUTPUT"
    elif echo "$LABELS" | grep -q '"release:minor"'; then echo "level=minor" >> "$GITHUB_OUTPUT"
    else                                                  echo "level=patch" >> "$GITHUB_OUTPUT"
    fi
```

The PR-title-fallback path in the acceptance list (`release: minor — ...`) is implemented as a second branch when no `release:*` label is present but the title matches `^release:\s*(major|minor|patch)\b`. This is a defensive fallback for the case where a contributor wrote the convention into the title but forgot to add the label — the workflow still won't fire (the `if:` requires a label) so the title fallback is in fact only useful for **`changes` job's e2e detection**, not for `bump.yml`. Documenting the asymmetry: the bump is gated on labels (a deliberate human ack), the e2e is gated on either label OR title prefix (a softer signal that catches in-flight release PRs before the label is added).

### Version bump + tag + push

```yaml
- name: Configure git identity
  run: |
    git config user.name "github-actions[bot]"
    git config user.email "41898282+github-actions[bot]@users.noreply.github.com"

- id: bump
  run: |
    NEW=$(npm version ${{ steps.level.outputs.level }} -m "chore(release): %s")
    echo "version=$NEW" >> "$GITHUB_OUTPUT"

- name: Push commit + tag
  run: git push origin main --follow-tags
```

`npm version` does three things: edits `package.json` `version`, creates a commit, creates a tag (`v<X.Y.Z>` by default). `--follow-tags` pushes both. Output `version` is the literal string (`v0.2.0`) returned by `npm version`.

**Branch-protection caveat:** `main` is protected (per the AGENTS.md note about the `Test` check rule). `GITHUB_TOKEN`-pushed commits will be rejected by branch protection unless GitHub Actions is granted a bypass under repo Settings → Rules → Bypass list, OR the rule allows the `github-actions[bot]` actor. This is a one-time repo-config step, not something I can change from a PR. Documented in the PR body so the reviewer can flip the toggle before merging.

### Draft GitHub release

```yaml
- env: { GH_TOKEN: ${{ secrets.GITHUB_TOKEN }} }
  run: |
    gh release create "${{ steps.bump.outputs.version }}" \
      --draft \
      --generate-notes \
      --title "${{ steps.bump.outputs.version }}"
```

`--draft` keeps the human gate — `publish.yml` triggers on `release: published`, so the npm push doesn't fire until a human clicks **Publish release** in the GitHub UI.

## Half B — `ci.yml` e2e gating (modify in place)

### Add an e2e-required output to the `changes` job

The `changes` job already runs unconditionally and computes `code` (paths-filter). Add a sibling step + output:

```yaml
outputs:
  code: ${{ steps.filter.outputs.code }}
  e2e:  ${{ steps.e2e-required.outputs.required }}
steps:
  ...existing paths-filter step...
  - id: e2e-required
    env:
      EVENT:  ${{ github.event_name }}
      LABELS: ${{ toJSON(github.event.pull_request.labels.*.name) }}
      TITLE:  ${{ github.event.pull_request.title }}
    run: |
      if [ "$EVENT" = "push" ]; then echo "required=true"  >> "$GITHUB_OUTPUT"; exit 0; fi
      if echo "$LABELS" | grep -qE '"release:(major|minor|patch)"'; then echo "required=true"  >> "$GITHUB_OUTPUT"; exit 0; fi
      if echo "$TITLE"  | grep -qiE '^release:[[:space:]]*(major|minor|patch)\b'; then echo "required=true"  >> "$GITHUB_OUTPUT"; exit 0; fi
      echo "required=false" >> "$GITHUB_OUTPUT"
```

Three branches in precedence: push-to-main → always required; label match → required; title-prefix fallback → required; else → not required.

### Gate the test matrix

Current test-job condition: `if: github.event_name == 'push' || needs.changes.outputs.code == 'true'`.

New: shard 1 still runs whenever there's code (for unit tests), shards 2-4 only run when e2e is required. One job-level `if:`:

```yaml
if: |
  github.event_name == 'push' ||
  (needs.changes.outputs.code == 'true' &&
    (matrix.shard == 1 || needs.changes.outputs.e2e == 'true'))
```

GitHub evaluates job-level `if:` per matrix cell, so cells 2-4 cleanly drop on non-release PRs while cell 1 still runs.

### Gate the e2e step inside the test job

Cell 1 still runs by the matrix gate above (because of the `matrix.shard == 1` clause), but on a non-release PR cell 1 should run **only** the unit tests, not the shard-1/4 e2e slice. Add a step-level `if:` to the existing `E2E` step:

```yaml
- name: E2E
  if: github.event_name == 'push' || needs.changes.outputs.e2e == 'true'
  run: bun run test:e2e -- --shard=${{ matrix.shard }}/4
```

Result table:

| Scenario | Shard 1 | Shards 2-4 | Unit | E2E |
| --- | --- | --- | --- | --- |
| Push to main | runs | runs | ✓ | ✓ all 4 shards |
| PR, code change, `release:*` label | runs | runs | ✓ | ✓ all 4 shards |
| PR, code change, no label | runs | skipped | ✓ | skipped |
| PR, doc-only (no `code` change) | skipped | skipped | – | – |

### Aggregator stays as-is

`test-aggregator: needs: [test]` already treats `result == 'skipped'` as success and asserts `success` otherwise. Matrix-level skipped cells fold into the aggregated `needs.test.result`: GitHub returns `success` if all non-skipped cells succeeded, `skipped` if all cells were skipped (doc-only case), `failure` if any failed. The aggregator's existing branches cover both. No change needed.

### What stays running on every PR

- **`secrets`** (Secret Scan) — unchanged, no `if:`, runs every PR + push.
- **`lint`** (Lint & Format + typecheck + security pattern check + CLAUDE.md presence check) — unchanged, no `if:`, runs every PR + push. Satisfies "Unit tests + lint + security scan continue to run on every PR".
- **`build`** — `needs: [lint, test]`. On non-release PR, `test` is `success` (shard 1 unit-only), so `build` runs. On doc-only PR, `test` is `skipped`, so `build` is also `skipped` (current behavior).

## Required label setup (out-of-PR action)

The repo's existing labels (`bug`, `documentation`, `duplicate`, `enhancement`, `good first issue`, `help wanted`, `invalid`, `phase-1`, `question`, `wontfix`) do not include `release:*`. The labels need to be created once before the workflows can recognize them:

```bash
gh label create release:major --color B60205 --description "Bump major version + draft release on merge"
gh label create release:minor --color FBCA04 --description "Bump minor version + draft release on merge"
gh label create release:patch --color 0E8A16 --description "Bump patch version + draft release on merge"
```

This is a one-time repo-config step (GitHub doesn't auto-create labels referenced from workflows). Documented in the PR body for the reviewer.

## Branch-protection caveat (out-of-PR action)

For `bump.yml` to push to `main`, GitHub Actions needs a bypass on the branch-protection rule. Repo Settings → Rules → Bypass list → add **GitHub Actions** (or the `github-actions[bot]` user). Without it, the push step will fail with `protected branch`.

Alternative: switch the bump flow to "open a PR with the version bump" instead of pushing directly. Rejected here because it adds a human-merge step, defeating the automation goal — the whole point is "label your PR and the next merge ships it."

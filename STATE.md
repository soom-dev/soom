## Recent Decisions Worth Knowing

These are the gotchas and architecture choices from the last few PRs that are most likely to bite if you touch related surfaces. Treat them as project-culture notes, not exhaustive documentation.

- **A modal-dialog a11y contract has six independent load-bearing pieces — implementing only some of them ships a broken affordance.** PR #47 added the help dialog and the implementation surface had to land all of: (1) `role="dialog"` + `aria-modal="true"` so AT recognizes the dialog and treats outside content as suppressed; (2) `aria-labelledby` referencing a heading *inside* the dialog so it announces with a name; (3) `aria-hidden` toggled on open/close so AT visibility tracks the dialog state independently of CSS; (4) focus moves to a child of the dialog on open (otherwise keyboard users press the trigger and stay focused on the now-obscured trigger, then Tab walks behind the dialog); (5) Tab is trapped inside the dialog (Tab from last → first; Shift+Tab from first → last); (6) focus is restored to the invoking element on close, including on Esc and backdrop dismiss (the click-the-close-button path works "by accident" because focus naturally lands there before close — Esc/backdrop is where the breakage shows up). Three of the six (focus moves on open, trap holds on Tab, focus restored on close) cannot be validated by unit tests on rendered HTML alone — they require a live DOM probe with active-element tracking. **For any future modal-dialog work in this codebase, run a 7-state Chromium probe modeled on `/tmp/verify-help.mjs` (initial → open via click → Tab × 2 → Esc → open via shortcut from a different element → backdrop click) before claiming completion.** Pattern generalizes to any "summons-then-dismisses-with-restoration" UI (popovers, tooltips with focus, bottom-sheets) — not just dialogs strictly speaking.
- **Theme tokens decompose into three buckets — *anchor* (must shift per-mode), *derived* (mechanically follows from anchors), *theme-neutral* (identical across modes) — and only the first two should change between themes. When deepening a sibling color for a different ground, derive by HSL lightness shift, not by RGB hand-tweak or alpha-stacking.** PR #51 rebased the light theme onto Sapphire Nightfall: 5 anchor tokens (bg, accent, edge-color, completed-fill, subgraph-border) shifted explicitly with documented contrast math; ~10 derived tokens (annot-bg, annot-text, annot-border, accent-glow, edge-glow, controls-bg, subgraph-bg, subgraph-nested-bg, etc.) followed mechanically from `bg + alpha`, `accent + 20% alpha`, etc.; ~7 theme-neutral tokens (label color, dormant edge stroke `rgba(26,26,46,0.4)` reads soft-gray on either ground, shadow alphas, marker fill, grid dot, edge-shadow alpha) preserved across modes. The anchor-vs-derived distinction matters because if a derived token doesn't visually match its rule, the rule is wrong, not the token. **HSL derivation:** dark accent `#0A7BC4` (HSL 202°, 90%, 40%) → light accent `#0A5A8C` (HSL 204°, 87%, 29%) — keep H near-constant, keep S near-constant, shift L. The pair reads as "one color under different lighting," which is exactly the brand goal. Hand-tweaking RGB to "darken" usually shifts hue accidentally because channels don't scale proportionally. Alpha-stacking a single color over different bgs breaks the visual identity because alpha blends differently against each ground. **Mermaid v11 leak diagnostic:** when a "the new theme has wrong colors" report comes in, first triage is `grep -oE "fill:#[0-9a-fA-F]{3,8}" /tmp/render.html | sort -u` — anything in the Flat UI palette (`#1abc9c #2ecc71 #4a90d9 #9b59b6 #e74c3c #f39c12`) is Mermaid's emitted SVG `<style>` rule winning on specificity, not the soom theme layer. Out of scope for theme PRs; would need post-processing on the emitted SVG.
- **A shared const should describe a *contract*, not a string-template family. Three diagnostics decide whether a candidate belongs: matching geometry (or whatever value-axis the const captures), matching theme-token family (or the shared resource the values reference), AND matching call-site shape (TS module consumer vs CSS rule vs computed-style assignment). If any of the three differs, leave the candidate inline — and document the exclusion explicitly so the boundary doesn't erode under "I'll just clean this up while I'm here" PRs.** PR #50 centralized 4 declarations + 5 usages of node-elevation shadows into `SHADOW = { rest, active, completed }` in `src/constants.ts`. Two other `drop-shadow(...)` strings in `themes/base.ts` looked like they belonged but didn't: the edge ambient shadow uses different geometry (`1px 2px 3px` vs node `-4px 6px 10px`), a different theme-token family (`--soom-edge-shadow` vs `--soom-shadow-rest/active/completed`), and a different call-site shape (static CSS rule vs `timeline.add({ filter: [...] })` tween target); the flow-particle glow differs on all three axes again. Collapsing them into `SHADOW` would re-collide what CSS variable namespacing had already separated. **Audit pattern for any "centralize N places" backlog item:** before scoping the work, run TWO greps — one for declarations, one for usages — and report both counts (they usually disagree by 1.5–3×; the disagreement tells you whether the const is decl-heavy or usage-heavy). PR #50's backlog said "5 places"; reality was 4 decls + 5 call sites + 2 unrelated inline strings that grep-matched but didn't belong. **Documentation rule:** when centralizing N-of-N+M sites, the canonical-source-of-truth doc must call out the M excluded sites by name with a positive-framed reason ("X is a separate visual language because…"), not a negative one ("not added"). Without that, future contributors read the boundary as accidental, not deliberate. Same shape applies to any future centralization — `TIMING`, `Z_INDEX`, `LAYOUT` already in `src/constants.ts`, and any future tokenization (`TYPOGRAPHY`, `BREAKPOINTS`, etc.) that comes up.
- **For chrome whose visibility flips on auto-hide, prefer CSS-only via the existing `.soom-controls-hidden` body class over a JS toggle in the auto-hide handler — and audit the runtime API surface before extending it.** PR #48's progress pin lives entirely under `.soom-controls-hidden #soom-progress-pin { display: block; }` — no new JS state, no toggle calls in `showControls()`/`hideTimer`. Watermark `bottom: 16px ↔ 60px` (PR #45) and annotation `bottom: 80px ↔ 128px` already use the same pattern. The body class is the single source of truth; piggy-backing means new chrome can never drift out of sync with the rest during the auto-hide transition. Audit pattern: `grep '\.soom-controls-hidden' src/themes/base.ts` shows what's already scoped before adding more. **Runtime-API corollary:** PR #48's DoD listed `src/runtime/api.ts` as a permitted edit site, but `api.progress` was already on the public `SoomAnimationApi` surface (line 163–165) and the existing `rafLoop` in `controls.ts` was the natural call site for per-frame UI updates. Before reaching into the runtime for a new consumer feature, grep `SoomAnimationApi` for the data the feature needs — if it's already on `window.soomAnimation`, the work lives entirely in `src/output/`. Same lesson inverted from PR #38's `setSpeed` debacle (assignment to `playbackRate` was silently dropped because the runtime *didn't* expose it as a writable property): the public contract is the source of truth either way.
- **The visual gate is structurally blind to six distinct classes of UI drift: sub-region pixel changes diluted under `maxDiffPixelRatio: 0.02` (PR #39, dormant-edge contrast), styling on transient runtime classes that never appear in a settled-state snapshot (PR #40, `.soom-node-active`), newly-introduced sub-2% UI components (PR #44, annotation panel ≈ 1.6% of viewport), repositioning + resizing + reopacity of an existing UI component below the threshold (PR #45, watermark moved 408×61 → 120×28 — total pixel diff ≈ 0.55%), chrome scoped under `.soom-controls-hidden` that the capture path never enters (PR #48, progress pin — `visual.spec.ts` calls `pause()` then `goToStep()`, and `showControls()` returns early when paused, so the body class never gets set during capture), AND any change to a non-default theme — `visual.spec.ts:30` runs `bun run src/cli.ts render ${mmd} -o ${html}` with no `-t` flag, so the suite is **dark-theme-only** and light-mode token shifts (PR #51, full sapphire rebase) never reach the pixel diff.** Direction-agnostic: making something more visible (PR #39), changing transient styling (PR #40), adding a small visible component (PR #44), *moving* a small component (PR #45), *adding auto-hide-only chrome* (PR #48), and *changing a non-default theme entirely* (PR #51) all slide under the noise floor or out of capture entirely. **Coverage check pattern:** before assuming "visual baselines will fail," `find tests -name "visual.*"` + `grep -n "render\|theme" visual.spec.ts` is the 30-second precheck that confirms which paths are actually exercised. The DoD's claim of "visual tests will fail" can be overstated by audits that assume more comprehensive baseline coverage than actually exists. Validate accessibility / visibility / placement / palette contracts through unit tests against `baseCss` text + computed-style assertions in real browser (`getComputedStyle(el)`, `el.getBoundingClientRect()`, `el.getAttribute(...)`), not screenshots. The gate's job is "did the scene structure change unexpectedly *in dark theme at default capture conditions*," nothing more. Repeatable verification pattern: a one-shot `bun /tmp/verify-<component>.mjs` that launches Chromium with the relevant theme + settled state, dumps computed styles + bounding rects + attributes. Three layers (visual gate / unit tests / out-of-band browser inspection) together pin the contract; any one alone is incomplete. *Audit gap to fix later:* light-mode visual coverage — after PR #51 light is a first-class brand variant, not a "derivative."
